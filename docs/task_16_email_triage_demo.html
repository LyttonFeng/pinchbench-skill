Baseline vs RL LoRA：邮件 triage 能力对比

这个页面展示同一个任务在原始 Qwen3-4B 和 RL LoRA 下的表现差异。 重点不是“模型会不会回答”，而是它会不会真正把邮件读完、排好优先级、并输出可交付的 triage 结果。

适合做汇报 / 方案展示 / 手机查看

任务说明

输入是什么，输出要什么
task_16_email_triage

任务目标

  • 读取全部邮件
  • 识别重要客户 / 高优先级事项
  • 按优先级分类 triage
  • 输出最终处理结果

看点

  • 是否会先完整读取再判断
  • 是否能稳定排序而不是乱分
  • 是否输出可交付的 triage report
  • 是否能把任务真正做完

Baseline 起点

Qwen3-4B 原始模型
任务完成度39%
常见表现：能开始 triage，但优先级判断不稳，容易漏重点。
问题：知道要做什么，但执行链路不够稳定。

RL LoRA 结果

Qwen3-4B + LoRA
任务完成度89%
变化：更稳定地完成分类、排序和输出，triage 结果更完整。
结论：RL 让模型不只是“会说”，而是更会把事情做完。

Baseline 过程 原始 Qwen3-4B

Urgent: client escalation on invoice issue
Needs attention today
P1
FYI: weekly newsletter
No action needed
P2
Contract renewal from big client
High value, should prioritize
P0
观察：baseline 多步也会走，但常见问题是优先级判断不稳，或者中途就把 triage 结论提前收掉。
重点：它不是“完全不会做”，而是多步执行链没把任务稳稳做完。

LoRA 过程 RL 训练后

Contract renewal from big client
P0 · prioritize first
P0
Urgent: client escalation on invoice issue
P1 · follow-up required
P1
FYI: weekly newsletter
P2 · low priority
P2
观察：LoRA 更稳定地推进多步执行，分类、排序和输出更完整，triage 更像真正交付结果。
重点：RL 把“每一步都继续推进任务”的行为学稳了。

多步执行轨迹

模型 action -> 环境反馈 -> 下一步 action
重点展示“过程”而不是只看最终分数

Baseline 轨迹 更容易早停 / 漏优先级

Turn 1Model action
模型动作
先粗看邮件标题，尝试直接给出分类结论。
→
环境反馈
发现高优先级客户邮件还没完整确认，triage 结果不够稳。
Turn 2Model action
下一步动作
继续尝试分类，但优先级顺序没有明显修正，容易漏掉重点邮件。
→
环境反馈
任务推进有限，输出内容可读但不够完整，结果得分偏低。
Turn 3End state
收尾动作
模型提前结束，给出了一个“能看但不够完整”的 triage report。
→
环境反馈
邮件 triage 没有充分覆盖全部重点，最终完成度停在 baseline 水平。
关键信号：模型知道要做 triage，但没有把“先完整读、再稳定排优先级、再交付”跑通。

RL LoRA 轨迹 更会推进任务

Page 1 / Turn 1先完整扫描邮件
模型 action
完整扫描邮件列表，先抓高优先级客户和当天要处理的事项。
→
环境反馈
系统返回更清楚的任务状态，过程 reward 开始变正。
Page 2 / Turn 2继续补全分类
下一步 action
根据前一轮反馈，补齐 P0 / P1 / P2 分类，保持 triage 结构完整。
→
环境反馈
triage 更稳定，输出开始接近 benchmark 期待的格式。
Page 3 / Turn 3完成交付
收尾 action
完成 triage report，输出结构清楚、重点明确的最终结果。
→
环境反馈
任务完成度更高，结果更像一个真正可交付的邮件 triage 产物。
关键信号：RL 不只是让结果更好，而是让“多步执行链”更稳，最终能把任务做完。

结论

这个 demo 想表达什么
1
Baseline 能做，但不够稳 会进入任务，但优先级判断和交付结果不稳定。
2
RL 之后更会推进任务 模型更稳定地读完邮件、分类、排序并输出结果。
3
提升的是任务完成能力，不只是回答质量 这个 demo 适合用来说明 RL 对真实 agent 任务的作用。
