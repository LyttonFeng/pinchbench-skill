Slide 1 · Task

这个任务到底在测什么？

不是“读邮件分类”，而是做一轮值班 inbox triage：读完 13 封杂乱邮件，识别核心事件，给出优先级，并写成一份可执行报告。

任务要求

读完整个 inbox， 写出 triage_report.md
  • 覆盖全部 13 封邮件，不漏项
  • 为每封邮件给出 priority / category / action
  • 把结果组织成一份可扫描、可执行的值班报告

为什么难

Incident Linkage
Priority Propagation
Coverage
Closure
  • 不是单封分类：要看整个 inbox 的事件结构
  • 要把 `email_01` outage 和 `email_13` alert 连起来
  • 要稳定识别 BigClient / security 这类高权重邮件
  • 最后还要收敛成一份真正可用的报告

Slide 4 · Improvement

RL 到底把什么能力拉起来了？

不是“总结写得更顺”，而是三类核心能力明显更强：linkage、propagation、closure。

Before RL

Baseline
Incident Linkage 弱：相关邮件常被当成独立项
Priority Propagation 弱：BigClient / security 权重不稳
Closure 弱：输出更像邮件清单，不像 operator report
观感上像“逐封分类器”： - 每封邮件各看各的 - 关键邮件之间关系弱 - 结构有了，但值班感不强 - 结果可读，但不够可执行

After RL

Step30 LoRA
Incident Linkage 强：更容易把 outage 与 alert 归并成一个事件
Priority Propagation 强：BigClient / security 更稳定上浮
Closure 强：更容易写成结构化、可执行的最终报告
Automated: file_created=1.0, priorities=1.0, categories=1.0, actions=1.0, outage_is_p0=1.0, client_is_high_priority=1.0 Judge: contextual links are made; summary and priority ordering are clear. 结论：提升是结构性的，不是 cosmetic 的。
这不是“写得更顺”，而是 incident linkage + priority propagation + closure 变强了。

Slide 6 · Why Score Improved

这些变化，为什么会直接变成分数提升？

task16 的分数不是单一主观判断。它会把“产物完成度”“关键判断”“整体质量”一起算进去，所以结构变化会直接反映到得分。

Automated 分为什么会涨

写出了 triage_report.md 直接决定 `file_created`，而且没有最终报告时其余结构项通常也会一起崩。
覆盖更多关键邮件 会提升 `all_emails_covered`，尤其是 outage、BigClient、security、alert 这些关键项。
priority / category / action 更完整 直接映射到 `priorities_assigned`、`categories_assigned`、`actions_assigned`。
incident linkage 更稳定 会拉高 `alert_linked_to_outage`、`outage_is_p0`、`client_is_high_priority` 这些关键自动项。

LLM Judge 分为什么会涨

不再把邮件独立看待 会提升 `Contextual Awareness and Connections`，这是 task16 的核心能力之一。
高价值 / 高风险项被稳定上浮 会提升 `Priority Assignment Accuracy`，尤其是 BigClient 和 security 这类边缘但重要的邮件。
最终产物更像值班报告 会提升 `Report Structure and Usability`，因为 summary、day plan、priority ordering 更清楚。
行动建议更具体 会提升 `Action Recommendations`，从“respond”变成更像 operator 的可执行动作。
所以 `37.5% → 65.7%` 不是“评委主观觉得更好看”，而是 task 所关心的几个核心能力同时抬起来了。

Slide 7 · Graph

RL 激发出的能力：Inbox Graph

baseline 更像局部分类器；RL 后更像在整个 inbox 上建立 incident graph。

Baseline

节点大多孤立，关键邮件之间关系弱，优先级传播不稳定。

Local classification: `email_13` 没稳定并入 outage；`email_05` 和 `email_08` 更像独立事项，而不是在全局 inbox 中被重新加权。

RL Step30

关键节点之间连边更清楚，优先级从核心事件向相关邮件传播。

Global inbox reasoning: `email_01 ↔ email_13` 形成核心 incident group； `email_05` 和 `email_08` 被更稳定地上浮为高权重业务项。
RL did not just improve style; it improved operational judgment.
